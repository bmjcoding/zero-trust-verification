## 0. Position and posture (read first)

The index is a DERIVED VIEW, never a store of truth (ADR OWM). Every record carries a `source` = {repo, commit_sha, path, line?} back-pointer; the query surface returns the pointer with every answer so an agent (or human) opens the authoritative file. There is NO write-back, NO edit surface, NO second copy of any fact that outlives one crawl. Rebuild-from-empty is always correct; the index is a cache with a git-sha cache-key, so the staleness the repos already solve (PR Gate rot detection per ADR 0003, ADR supersession, manifest_revision) is inherited, not re-litigated.

Read-only / least-privilege posture matches the audit tier's low-trust adoption on-ramp (ADR 0011): OWM needs repo READ only, scoped to the declared memory globs (§ OWM-03a). It grants nothing an autonomous PR drain would; it has strictly LESS surface than the audit (which reads the whole tree).

## CORRECTION vs r1 (honesty about substrate — read before the reuse claims)

r1 asserted `scripts/host.sh` "already solves multi-repo enumeration and per-backend auth." That is FALSE and is retracted. Ground truth (verified against `plugins/autopilot/scripts/host.sh`, ADR 0013): host.sh is a 104-line PR/build PASS-THROUGH whose entire delegated surface is `pr-open pr-ready pr-state pr-comment pr-approve pr-decline pr-merge pr-merge-strategies build-status pr-list-ready` + the local `backend` introspection. It has NO repo enumeration, NO org listing, NO repo-content read; its per-backend auth is a single-repo PR-API credential, not an org-wide read token. It also lives INSIDE the autopilot plugin, not at repo-root `scripts/`. Consequences the hardened design honors:
>   (a) Org enumeration is a NEW capability class, not an existing one OWM inherits. It is added behind the adapter as a genuinely new backend method (OWM-09) — described as such, never as pre-solved.
>   (b) OWM must NOT depend on the autopilot plugin (that destroys independent installability — the ADR 0011/0001 failure class). It therefore treats the repo list as CONFIG-FIRST (OWM-03) and the host-enumeration path as an OPTIONAL enrichment the plugin degrades without; if org-enumeration is used it is vendored under the ADR 0001 byte-identity lint, never a runtime dependency on a sibling plugin.
>   (c) The enumeration credential's READ scope / blast radius and the cross-repo disclosure policy are ESCALATED (human_questions), not agent-assumed.

## A. Parse contract — the typed extractors

### OWM-01 — Memory-class parse contract: one typed extractor per memory-class [ADR OWM; CONTEXT.md]
The crawler recognizes exactly the memory-classes the suite already emits, each with its OWN small extractor (they do NOT share a frontmatter shape — a VERIFIED corpus quirk):
  - `adr`  — `docs/adr/NNNN-*.md`: H1 title on LINE 1, then a YAML frontmatter block AFTER the title (`---` on line 3), fields `status` ∈ {accepted, agent-decided}, `date`, optional `amended-by`/`superseded-by` (VERIFIED: all 15 repo ADRs use this post-title-frontmatter shape; `status: accepted` and `status: agent-decided` both occur). One decision per file (kebab-case stable name = stable ID).
  - `manifest` — Verification Manifest YAML: parsed by the EXISTING scripts/validate_manifest.py path (ADR 0014) — reuse, never re-implement. Harvest schema_version, manifest_revision, spec.path/title/spec_hash, journeys (id, criticality ∈ CORE/SUPPORTING/DEV), behaviors (id, lifecycle). Validator exit contract is honored VERBATIM: exit 4 (schema-invalid) or exit 5 (unsupported version) → the file is NOT indexed as a manifest; it is indexed as `unparseable` carrying the validator's error + exit code (never dropped silently).
  - `glossary` — CONTEXT.md: `**Term**:` blocks → one term-definition record each, plus the `_Avoid_:` aliases (so a query for a rejected synonym resolves to the canonical term — directly serves the north star). (VERIFIED format: `**Term**:` on its own line, definition prose, `_Avoid_: alias, alias` line.)
  - `decision-log` — greppable `DL-###` one-line entries in PR bodies, trackers, and manifest `interrogation.log` (the format ADR 0002 + CONTEXT.md pin). Harvested as append-only events keyed by (repo, sha).
  - `journey` / `as-built` — journey maps + as-built docs referenced by manifests; indexed as PROSE records with their `source` pointer (retrieval, not structural claims).
**Non-goal:** OWM does not DEFINE any of these formats. It vendors the recognizers from the single canonical sources (the manifest JSON Schema; behavior-coverage-format.md; the ADR frontmatter convention) — a format change is a cross-plugin change caught by OWM-10.
**Acceptance:** `[det]` `scripts/extract_memory.sh <file>` emits one normalized JSON record set per class; golden-fixture matrix per class incl. the ADR post-title-frontmatter quirk, a schema-invalid manifest (→ `unparseable` + validator error + exit code, NOT dropped), a `status: agent-decided` ADR, an `_Avoid_` alias, and a `superseded-by` ADR. `[drain]` extractor RECALL on the messy long tail of real-world file variants (agent-judged residual — the fixtures prove the KNOWN shapes only).

### OWM-02 — Record schema + stable cross-repo IDs [ADR OWM; ADR 0014 pattern]
ONE JSON Schema (`schema/org-memory/v1.schema.json`) is the sole structural source of truth for an index record (mirrors ADR 0014's single-schema discipline): required `{kind, id, org_id, repo, commit_sha, path, source_line?, title, status?, date?, body_excerpt, aliases[], supersedes?, superseded_by?, spec_hash?}`. `org_id` = stable cross-repo identity = `<repo-slug>:<kind>:<kebab-name>` (ADRs / glossary terms / behavior IDs already have stable kebab names — reuse them; nothing minted). schema_version pinned; validated by the same jsonschema Draft validator (ADR 0014). Additive-only evolution rule inherited verbatim from the manifest schema: unknown fields ignored, major-version bump refuses (forces index rebuild).
**Acceptance:** `[det]` record instances validated by the REUSED validator; fixture proves an id collision across two repos is DISAMBIGUATED by `repo` (not merged), and a same-repo same-name ADR revision is treated as supersession not duplication.

## B. The crawler

### OWM-03 — Crawler over a CONFIG-FIRST repo list [ADR 0011 — independent installability]
The crawler takes a repo list. PRIMARY source = explicit config (a checked-in repo-list file / plugin config) — this is the DEFAULT and requires no host-enumeration capability, so OWM installs and runs standalone without the autopilot plugin. OPTIONAL enrichment = host-side org enumeration (OWM-09), used only when configured and available. For each repo at a pinned ref, the crawler walks ONLY the declared memory globs (§ OWM-03a) applying OWM-01 extractors. Per-repo failure is isolated (one unparseable repo never fails the crawl; recorded as a `crawl_error` record with repo+sha, surfaced by OWM-08). Idempotent: same (repo, sha) → same records.
**Acceptance:** `[det]` `scripts/crawl.sh --repos <fixture-dir-list>` against a fixture tree of ≥3 mini-repos produces a deterministic record set (golden JSON, byte-stable under LC_ALL=C sort); a planted unparseable file yields a `crawl_error` record and a still-green crawl of the siblings; a run with ZERO host-enumeration capability configured completes from the explicit config list alone (independent-installability proof).

### OWM-03a — HARD-BOUNDED read surface: memory globs only, NEVER a whole-repo walk [NEW — feasibility/cost guard]
The crawler reads ONLY an explicit, closed allow-list of memory paths: `docs/adr/*.md`, `CONTEXT.md`, the manifest glob, the journey/as-built globs, and the decision-log carriers named in OWM-01. It MUST NOT read the code tree, MUST NOT recurse arbitrary directories, and MUST NOT grep every file in a repo. This is the org-memory analog of the codebase-health invariant that bounded probes never run the whole suite: an unbounded per-repo tree walk across an org is the cost/perf failure this item exists to prevent. A per-repo ceiling (file count + total bytes over the memory globs) is enforced; exceeding it yields a `crawl_error` (`memory-surface-oversized`), never a silent partial-then-timeout. Reading anything outside the allow-list is a bug the self_test catches.
**Acceptance:** `[det]` a fixture repo containing a large code tree PLUS the memory globs → the crawl record set references ONLY files under the declared globs (a golden assertion that no code-tree path appears in any record or in the crawler's opened-file trace); a fixture whose memory surface exceeds the ceiling yields the `memory-surface-oversized` crawl_error, not a hang. `[drain]` real per-repo crawl latency on a large live corpus (measured, not asserted).

### OWM-04 — Incremental crawl keyed by commit sha [ADR OWM §0]
The index stores, per repo, the last-crawled commit_sha. A re-crawl seeing an unchanged head is a NO-OP (the git-sha cache-key); a changed head re-extracts only that repo. This is the ENTIRE staleness story: repos own freshness (PR Gate rot, manifest_revision, ADR supersession); OWM only mirrors the current committed bytes. There is deliberately NO independent invalidation heuristic to drift.
**Acceptance:** `[det]` two crawls over the same fixture sha → second is a proven no-op (record set + a `SKIPPED unchanged` note); a single-file change in one repo re-extracts ONLY that repo (others untouched, proven by unchanged record hashes).

## C. The index + query

### OWM-05 — Index store: SQLite (single-file, uv-run Python) [ADR 0015 — reuse toolchain]
The index is a single SQLite file built by Python-on-uv (no new language runtime; ADR 0015). Chosen over a flat JSON blob (full-scan per query) and over any server DB (an ops dependency that contradicts "derived cache, rebuild-from-empty"). Schema: one `records` table mirroring OWM-02 + FTS5 virtual table for full-text. The file is disposable — `rebuild` drops and re-crawls. NON-GOAL: no ORM, no migrations framework; a schema_version change gates a full rebuild.
**Acceptance:** `[det]` `scripts/index_build.sh <records.jsonl> <out.db>` produces a queryable db; a rebuild from the same input is byte-comparable on a canonical dump; an FTS query returns the seeded record WITH its `source` pointer.

### OWM-06 — Query surface: exact-lookup + full-text + alias-resolution [north star]
Query primitives over the index: (a) `lookup <org_id>` → the record + source pointer; (b) `search <text>` → FTS5-ranked records; (c) `resolve <term-or-alias>` → canonical glossary term (an `_Avoid_` synonym resolves to the real term — the "told twice" killer); (d) `decisions <topic>` → decision-log + ADR records for a subject, supersession-aware (a superseded ADR is returned FLAGGED, never as live truth). Every result carries `{repo, commit_sha, path, source_line}`.
**Acceptance (r2 — the `[det]`/`[drain]` line redrawn so no ranking is laundered as deterministic):**
  `[det]` (MECHANICAL only): (i) `resolve` maps a fixture `_Avoid_` alias to its exact canonical term; (ii) a superseded ADR is returned carrying its `superseded_by` flag and is NEVER returned as the current/live record (exact flag+field assertion); (iii) `lookup <org_id>` returns exactly the seeded record; (iv) every result JSON carries a NON-EMPTY source pointer; (v) supersession-awareness: given a fixture ADR pair, `decisions` returns the superseding one unflagged and the superseded one flagged.
  `[drain]` (RANKING QUALITY — NOT deterministic, never `[det]`): `search`/`decisions` relevance ordering ("does the right ADR come first on a real corpus") is measured against a live corpus, never asserted. r1 blurred "cross-repo search ranks the on-topic record" into `[det]`; that is retracted — ranking is `[drain]`.

### OWM-07 — Freshness metadata on every answer (staleness disclosed, never hidden) [ADR OWM §0]
Every query result includes `indexed_at_sha` for its repo + the crawl timestamp. If the repo's current head is knowable cheaply (config flag; else omitted), a `head_sha` and boolean `possibly_stale` (indexed sha != head). OWM never asserts freshness it cannot prove — `possibly_stale` is UNKNOWN-safe (ABSENT when head is unknown), never a false "fresh".
**Acceptance:** `[det]` a result over a fixture whose head advanced past the indexed sha carries `possibly_stale: true`; a result where head is unknown OMITS the field (no false-fresh); a same-sha result carries `possibly_stale: false`.

### OWM-08 — Coverage + crawl-error report (the "what do we NOT know" surface)
`scripts/coverage.sh` emits, per org: repos crawled, records by kind, and every `crawl_error` (unparseable file, unreachable repo, schema-invalid manifest via exit 4/5, `memory-surface-oversized`) WITH its source. This is the honest edge — an agent asking a question the index cannot answer sees WHY (repo not crawled / file unparseable / surface oversized), not a silent empty result. Mirrors codebase-health's measured-coverage honesty (`[det]` self-test half + `[drain]`/audit-run blind half).
**Acceptance:** `[det]` a fixture org with one unparseable ADR and one manifest failing schema validation (exit 4) yields a coverage report naming both with their paths + exit code; a fully-clean org reports zero errors.

## D. Consumption + packaging

### OWM-09 — host.sh gains a NEW `repo-list` backend method — org enumeration is a NEW capability, not a pre-solved one [ADR 0013 — extend the adapter; OPTIONAL]
Multi-repo discovery is a host concern, so IF built it belongs BEHIND the existing adapter as a genuinely NEW backend method — NOT smuggled as "already solved." `host.sh repo-list [--org <org>]` is added to the delegated subcommand surface; both backends (GitHub `gh`, Bitbucket DC REST) implement it against the SAME T01-class contract-test matrix (ADR 0013: a new capability is a backend method passing the existing matrix, never a new caller path). This is a change to the autopilot-resident host.sh, so it lands as an autopilot-tier register item OR the enumeration path is vendored into OWM under the ADR 0001 byte-identity lint (OWM-10) — OWM does NOT take a runtime dependency on the autopilot plugin (ADR 0011 independent-installability). OWM never touches a host API directly; if org-enumeration auth is unavailable OWM falls back to the OWM-03 explicit config list (degrade gracefully, ADR 0001 spirit) — the build never blocks on the credential decision. The enumeration credential's READ scope + cross-repo disclosure policy are ESCALATED (human_questions), not agent-assumed.
**Acceptance:** `[det]` `repo-list` added to the T01-class mock matrix for BOTH backends (mock org → TSV of repos); OWM crawler consumes the mock output; explicit-list fallback fixture (no `repo-list` capability → uses config list, loud note). `[drain]` real org enumeration + live credential path against a real host (per-backend property behind host.sh — the mocks prove the contract, not the live credential).

### OWM-10 — Vendoring lint: OWM recognizers pinned to the canonical formats [ADR 0001 — extend lint_consistency.sh]
OWM's extractors depend on formats owned elsewhere (manifest JSON Schema; `## Behavior coverage` format; the ADR frontmatter convention). A new `lint_consistency.sh` rule (V7) asserts OWM's manifest-parse path uses the CANONICAL validator/schema (not a forked copy) and that OWM's format-recognizer references name the single-source docs — so a manifest-format change can never leave OWM silently parsing the old shape. If the OWM-09 enumeration path is vendored, V7 also byte-identity-pins that vendored copy (no runtime autopilot dependency). Extends the existing lint; no parallel lint infrastructure.
**Acceptance:** `[det]` new V7 rule GREEN on the real tree; a planted OWM recognizer that FORKS the manifest schema instead of reusing scripts/validate_manifest.py is caught RED (added to suite_self_test.sh's planted-drift matrix, with a false-positive guard for a benign reference reword).

### OWM-11 — MCP server: the agent query surface, refuse-by-default on out-of-scope repos [north star; ADR 0011 read-only posture]
An MCP server (Python-on-uv; ADR 0015) exposes OWM-06 primitives as read-only tools (`memory_lookup`, `memory_search`, `memory_resolve_term`, `memory_decisions`) + one resource for coverage (OWM-08). It is a THIN adapter over `scripts/query.sh` — the deterministic query surface is the source of truth; the MCP server never re-implements retrieval (same discipline as validate_manifest.sh wrapping the .py). READ-ONLY tools; NO mutation surface. Two loop-safety / trust invariants (r2 additions):
>   (a) REFUSE-BY-DEFAULT scope: the tools serve ONLY records from repos in the configured allow-list; a query that would surface a repo outside the configured scope returns an explicit refusal + the reason, never the record. This is the enforcement point for the cross-repo disclosure policy that is ESCALATED (human_questions) — the policy is a human call; the refuse-by-default default is the safe agent-decidable posture until it is answered.
>   (b) SELF-EXCLUSION: OWM never indexes its own emitted artifacts (coverage reports, any file it writes) — the crawler's memory globs exclude OWM's own output paths, so no citation loop where org-memory re-ingests and re-serves its own derived view can form.
Ships in the plugin so an agent in ANY in-scope repo can ask the org what it already decided — the direct realization of "if an agent has to be told something twice, we've failed." Degrades gracefully: server absent → the CLI scripts still answer.
**Acceptance:** `[det]` an MCP protocol-level test (initialize + tools/list + a tools/call for `memory_lookup` against a seeded fixture db) returns the record + source pointer; the server is proven to shell out to `query.sh` (a fixture asserting IDENTICAL output between the tool and the CLI — no second retrieval impl); a query for a repo OUTSIDE the configured allow-list returns the explicit refusal, NOT the record (refuse-by-default proof); a fixture asserting OWM's own coverage-report path is EXCLUDED from the crawl (self-exclusion proof). `[drain]` a real agent reaching for the tools mid-session in a foreign repo (the north-star behavior is agent behavior — the test proves the tool answers, not that the agent asks).

## Non-goals
- **No second store of truth / no write-back.** OWM never edits, syncs, or authors a repo file. (ADR OWM — the whole point.)
- **No whole-repo read.** OWM reads ONLY the declared memory globs, never the code tree (OWM-03a). It has strictly LESS read surface than the audit tier.
- **No autonomous write / remediation loop / prod-triage / re-plan.** OWM produces NO findings that feed a spec-gen `--amend` or an autopilot re-plan; the remediation loop is reserved future scope elsewhere (spec-gen-tier-v1.md non-goals) and OWM does not build any part of it. No new autonomous write path exists, so infinite-regress / self-remediation is structurally absent, not merely guarded.
- **No mutation testing.** OWM runs no tests and mutates nothing; it neither runs nor ingests mutation reports (that is codebase-health's ingest-only concern, out of scope here).
- **No independent staleness heuristic.** Freshness = crawled sha vs head; no TTL, no semantic-staleness guess. Repos own rot (PR Gate, manifest_revision, ADR supersession).
- **No cross-repo write coordination** (Marshal's + claims' remit; ADRs 0009/0010/0011).
- **No new format ownership.** OWM recognizes existing formats; a format change is a cross-plugin change per OWM-10.
- **No runtime dependency on the autopilot plugin.** The host-enumeration path is config-first + optional + vendored-under-lint, never an install-time coupling (ADR 0011 independent installability).
- **No embeddings / vector store in v1.** FTS5 is the retrieval baseline; a semantic layer is deferred v2 (revisit trigger: FTS recall becomes a documented adoption blocker — mirrors ADR 0015's Rust-deferral discipline), and would sit BEHIND the query surface, never a new caller path.
- **No always-on service assumed.** The index is a rebuildable single-file cache; where/when it runs is an infra decision (human_questions), designed so OWM-01..08 never block on it.
