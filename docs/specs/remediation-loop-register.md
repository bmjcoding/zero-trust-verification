## A. Read the finding stream (no new emission)

### RL-01 — Finding-stream reader: `read_findings.sh <state.json> [--from-pr-gate <pr_gate_output>]`
Emits one normalized record per OPEN finding — `fingerprint | severity | slug | path | symbol | status | remediation_status` — from the `audit/state.json` findings map (and/or a captured `pr_gate.sh` run). **HARDENED (Defect A):** it does NOT emit an `expected_by` column — that field lives only in the fixture `EXPECTED_FINDINGS.yaml`, never in runtime `state.json` (VERIFIED: the runtime record carries `slug`, not `expected_by`). Deterministic provenance is derived downstream in RL-02 from the slug, not read as a stored field. Pure reader; NEVER writes state, NEVER re-runs the audit or a journey walk (loop-safety invariant 1; mirrors `pr_gate.sh`'s reporter-only posture). Degrade: missing/corrupt/unknown-`schema_version` state → emit nothing + loud `[note]`, exit 0 (invariant 4).
**Acceptance:** `[det]` fixture `state.json` (reuse the VERIFIED v2 shape) → asserted normalized rows; asserts NO `expected_by` column is fabricated; a corrupt-state fixture → zero rows + `[note]`, never a crash, never a guessed row.

## B. The two deterministic routers

### RL-02 — Eligibility gate: `finding_eligible.sh` (severity floor ∧ deterministic-provenance)
A finding is drain/escalate-eligible ONLY when severity ≥ floor (default HIGH — and HIGH already means CONFIRMED/traced per `severity-rubric.md`, so the floor is a confirmed-defect floor, not a suspicion floor) AND its slug is deterministically-scored. **HARDENED (Defect A):** provenance is NOT read from state.json; it is looked up in a lint-pinned `slug_provenance.tsv` (slug → `deterministic|agent`) seeded from SPEC_1.4.0 §12 (e.g. `dark-money-movement`, `log-only-refund`-seed, `memory-rot-dangling-ref`, markers, dead-code, giant-file, test-skip = deterministic; `non-idempotent-handler`, `unsafe-retry`, `missing-compensation`, `missing-dedup-guard`, misleading-name, semantic-drift = agent). Agent-scored slugs → INELIGIBLE(agent-evidence-only) (ADR 0004: no agent opinion auto-files rework). Floor from `remediation.config.yaml` (`severity_floor:`), default HIGH. **Unknown slug → INELIGIBLE(unknown-provenance)** — fail-SAFE toward inaction (invariant 4: broken/unknown → less action).
**Acceptance:** `[det]` truth-table: HIGH+det-slug → eligible; HIGH+agent-slug → ineligible(agent-evidence-only); MED+det-slug → ineligible(below-floor); config floor=MED flips the MED case; unknown slug → ineligible(unknown-provenance). `[det]` a lint (folds into V7) asserts every slug in `slug_provenance.tsv` exists in the audit taxonomy and its provenance matches SPEC_1.4.0 §12 (drift → red). Red-first per invariant 9.

### RL-03 — Escalate-class routing table (vendored, lint-pinned): `classify_fix.sh <slug>`
Maps an ELIGIBLE finding's slug → `DRAIN | ESCALATE` per ADR 0016. **drain-class** (reversible + gate-verifiable): `marker, dead-code, commented-code, suppression, memory-rot-dangling-ref, giant-file, test-skip, vacuous-test, missing-behavior-binding`. **escalate-class** (values/irreversible/external-fact, ADR 0002 trilist): the money/auth/tx slugs VERBATIM from `audit-state-and-verify.md` (`dark-money-movement, log-only-refund, non-idempotent-handler, missing-dedup-guard, unsafe-retry, double-submit-window, missing-compensation, missing-audit-trail`) + any `security/*` slug + wire-format/public-API-shape/alert-seam slugs. **Unknown slug → ESCALATE (fail-safe: never auto-drain an unclassified fix).** **HARDENED (Defect H — honest scope note):** because RL-02 already filters agent-scored slugs to INELIGIBLE, several escalate-class TX slugs (`non-idempotent-handler, unsafe-retry, missing-compensation` — agent per SPEC_1.4.0) never reach this classifier; for those the loop's actual behavior is INELIGIBLE→comment-only (ADR 0004-correct), NOT loop-escalation. The escalate-class table remains a superset (fail-safe if a slug is ever reclassified deterministic), but the register does NOT claim the loop autonomously escalates agent-scored money-path fixes — it correctly leaves them to human/audit review. Deterministically-scored escalate-class slugs (`dark-money-movement`, `log-only-refund`-seed) DO route ESCALATE.
**Acceptance:** `[det]` every listed slug returns its class; a money-path slug → ESCALATE; a marker slug → DRAIN; `--brand-new-slug` → ESCALATE (fail-safe, red-tested à la autopilot L13 non-vacuity). `[det]` lint V7 asserts the escalate-class table cites the ADR-0002 block and its money/auth membership is a SUPERSET of `audit-state-and-verify.md`'s TX slug catalog (drift → red).

## D. The two output paths

### RL-06 — DRAIN path: register → `/spec --from-findings` → GENERATE → Runbook PR + Story PR(s), none merged

So a drain-CLASSED finding that turns out to touch a vital/CORE path degrades to spec-gen's own S5 escalation (ADR 0008 degrade path) — the loop's slug-classifier is a first filter, spec-gen's refusal gate is the load-bearing second. The loop stamps each finding's `remediation` record (RL-08) SPEC_OPEN→PR_OPEN with the host PR ref. **Acceptance:** `[det]` mode handoff asserts STRAIGHT_THROUGH on a complete-manifest fixture and GENERATE_PAUSE on an incomplete one (reuses `detect_input_mode.sh` fixtures); branch-namespace + PR-ref stamping fixtured; asserts the loop opens NO merge call (Guard 3). `[det]` asserts the register bundling does NOT presume a single Story (no "one PR" assertion that would go red on a two-Story decomposition). `[drain]` end-to-end: spec-gen finalizing + autopilot draining (agent + real host).

### RL-07 — ESCALATE path: manifest-less GENERATE+pause → spec-gen S5, one question at a time

**Acceptance:** `[det]` an escalate-class finding produces a manifest-less input → `detect_input_mode.sh` returns NOT-STRAIGHT_THROUGH (GENERATE_PAUSE); the record→ESCALATED stamp fixtured. `[drain]` the S5 presenter surfacing one-at-a-time with dissent (agent).

## E. The three-guard ratchet (ADR 0017)

### RL-08 — Guard 1 (idempotency): additive `remediation` record on `state.json` + `already_filed.sh`
Additive `remediation: {status, ref, opened_at, remediation_depth}` per finding record (schema stays v2 — additive fields are not a break, VERIFIED precedent audit-state-and-verify.md:113-117). `already_filed.sh <fingerprint> <state.json>` → FILED when an open record (SPEC_OPEN|PR_OPEN|ESCALATED) or WONTFIX exists → skip LOUDLY (invariant 6, into Not-covered/`[skip]`). A human WONTFIX (existing `/verify --wontfix`, VERIFIED) permanently silences it for the loop too. **HARDENED:** the loop WRITES this `remediation` record — this is the ONE mutation the loop performs on `state.json`; it is idempotent and diffable (invariant 8) and additive-only (never repairs/guesses other fields, invariant 4). It does NOT set `status`/`verified_by` (those are `/audit`/`/verify`-owned).
**Acceptance:** `[det]` fixtures: PR_OPEN → FILED(skip-logged); WONTFIX → FILED(never-refiled); absent → UNFILED; the skip emits a `[skip]` line (asserted). `[det]` schema stays version 2 with the additive field (no-break assertion); `[det]` the loop touches ONLY the `remediation` sub-object (grep-assert no write to `status`/`severity`/`verified_by`).

### RL-09 — Guard 2 (depth ceiling): `remediation_depth.sh` — self-namespace → depth+1 → escalate
**HARDENED (Defect G — citation).** A finding whose branch traces to the loop's own `remediation/<slug>/*` namespace inherits `remediation_depth = parent+1`; RL-05 forces `depth ≥ 1 → ESCALATE` regardless of slug class. Bounds recursion at depth 1 by construction (a fix-of-a-fix always surfaces to a human). Reuses the VERIFIED `claim_overlap.sh --self-namespace <prefix>` → EXCLUDED primitive (claim_overlap.sh:26,107) — same own-namespace-detection mechanism, NOT a vaguely-cited "AV3-09".
**Acceptance:** `[det]` a finding on a `remediation/<slug>/*` branch → depth 1 → ESCALATE (even for a marker slug that would otherwise DRAIN); a finding on a normal branch → depth 0 → normal routing. Both fixtured, red-first, reusing the `--self-namespace` fixture style.

### RL-10 — Guard 3 (no tail-chasing) + tombstone-authored-by-spec-gen
**HARDENED (Defect C — the tombstone wiring bug).** Two inherited-free structural checks: (a) the loop NEVER merges — grep-assert no merge call in the drain path (autopilot HC4 already forbids it; this pins it at the loop layer). (b) When a drain-class removal deletes a symbol (dead-code/symbol delete), the `lifecycle: withdrawn` + `withdrawn_reason` tombstone that CH-05 depends on (VERIFIED `check_memory_rot.sh`) is authored by **spec-gen in the register session's manifest** — NOT by the autopilot Story PR, because spec-gen is the manifest's ONLY writer (VERIFIED HC3, MS §7). Concretely: the findings register for a removal declares the withdrawal as an acceptance behavior, spec-gen emits the tombstone into the manifest it owns, and the tombstone lands in the Spec/manifest PR of the drain; CH-05 then cannot flag the loop's own cleanup as memory-rot. **This corrects the draft's claim that the autopilot drain writes the tombstone — autopilot writes product code, never the manifest.** Cadence: unattended runs fire on the shared CI-cron pattern (Marshal/ambient-audit precedent); each fire is stateless (re-derives eligible findings from state.json + open PRs from `pr-list-ready`) so a missed/overlapping fire is harmless.
**Acceptance:** `[det]` a removal-register fixture produces a manifest with the `lifecycle: withdrawn` tombstone authored at the spec-gen stage (asserts the tombstone is in the manifest emission, NOT the impl diff); CH-05 emits NO rot finding for it (reuses CH-05's tombstone fixture). `[det]` grep-assert the drain path contains no merge invocation. `[drain]` stateless-fire idempotency across overlapping cron fires (real host).

## F. Mutation-testing / whole-repo guard (the user's explicit attack)

### RL-12 — No mutation-testing runs; no whole-repo probes: `remediation_scope_guard.sh`
**HARDENED (Defect E — mutation testing).** Loop-safety invariant 1 (VERIFIED): "Mutation-testing tools mutate the working tree, so no command, agent, or script ever runs one — their pre-existing reports are ingested like coverage," and probes are "never the whole suite." The remediation loop therefore: (a) NEVER triggers a mutation run — scoped OR whole-repo — under any code path; mutation-derived findings enter ONLY as pre-existing ingested reports via the normal audit stream (RL-01), exactly like coverage; (b) runs NO whole-repo scan of its own — it reads the ALREADY-COMPUTED `state.json` (diff-scoped at PR-Gate time, or the ambient audit's own output), never re-executing detection. The loop adds ZERO new probe/execution surface; its only compute is deterministic string routing over an existing state file. This directly answers "performance/cost scoping — it MUST NOT run whole-repo": the loop has no run-the-repo path to scope, because it re-uses the tiers' already-bounded outputs and executes nothing itself.
**Acceptance:** `[det]` a grep-assertion over the entire skill + RL-0x scripts that NO invocation of a mutation-testing tool (`mutmut|cosmic-ray|stryker|pitest|cargo-mutants`) and NO whole-repo `run_audit.sh`/full-suite call exists in any loop code path (red-tested by planting one). `[det]` asserts RL-01 only READS `state.json`, never spawns a detector (reuses the reporter-only posture assertion).

## Explicit non-goals (v1 of the loop)
- **A fifth plugin / new marketplace entry** — the loop is a codebase-health skill (ADR 0016); V6 stays exactly-four (VERIFIED).
- **New detectors or a new severity scale** — the loop reads the existing finding stream; authors no emission, holds no quality opinion (ADR 0016).
- **RUNNING mutation testing (scoped or whole-repo) or any whole-repo scan** — forbidden by loop-safety invariant 1; mutation findings enter only as ingested reports (RL-12). The loop executes NOTHING; it routes over an already-computed state file.
- **Auto-merging** — a Runbook PR + one-or-more Story PRs await human review, always (ADR 0007/0017 Guard 3; autopilot HC4). NOT "one Story PR" — the count follows the register's Story decomposition.
- **Auto-escalating agent-scored money-path findings** — agent-scored TX slugs (`non-idempotent-handler, unsafe-retry, missing-compensation`) are INELIGIBLE at RL-02 and stay comment-only (ADR 0004-correct); only deterministically-scored escalate-class slugs route to the loop's S5 escalation.
- **The loop writing manifest tombstones** — spec-gen authors them in the register session (it is the manifest's only writer, HC3); the loop never writes the manifest.
- **Depth > 1 auto-remediation** — a fix-of-a-fix always escalates (ADR 0017 Guard 2).
- **The long-running PR-event scheduler agent** — ADR 0003's separate future scope; this register specs the loop the scheduler would invoke, on the proven CI-cron/stateless cadence.
- **prod-telemetry → spec (the ops/triage agent)** — the second closed loop, deferred (three-tier-suite-consolidation memory item 2). Explicitly NOT in remediation-loop v1.
- **Reordering / prioritizing findings by agent judgment** — routing is deterministic (slug + severity); no importance ranking (mirrors Marshal's no-agent-judgment posture).
