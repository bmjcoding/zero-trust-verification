# ADR 0018 — The remediation loop is wiring, not a fifth checker; findings drain only when the FIX is deterministically-scored AND agent-decidable

---
status: agent-decided
date: 2026-07-06
---

The remediation loop (audit/PR-Gate/cron findings → Spec → drain → Story PR(s) awaiting human review) is built as **thin wiring over the three existing tiers**, exactly as ADR 0003 framed the long-running PR-review agent: "wiring, not a checker … it holds no quality logic of its own, so there is never a fourth opinion to keep consistent." It is NOT a fifth plugin. It adds no detector, no severity judgment, no second implementation of anything a tier owns. Concretely it is one skill (`/remediate`, homed in the codebase-health plugin at the VERIFIED path `codebase-health/plugins/codebase-health/skills/cleanup-audit/` — the tier that produces findings and owns `audit/state.json`, the ratchet, `severity-rubric.md`, and `loop-safety.md`) plus ~7 deterministic bridge scripts, covered by the existing plugin self-test.

**The pipeline, tier by tier (each seam already exists):**
1. **Producer — codebase-health.** The ambient audit (ADR 0003, CI cron) and the PR Gate (`pr_gate.sh`) already emit findings with a stable fingerprint (`sha1(path:symbol:slug)` first-12-hex, VERIFIED) and a severity from the one rubric (where HIGH already means confirmed/traced). No new emission is authored; the loop READS this stream.
2. **Selector — a deterministic bridge.** A finding is eligible to become work ONLY when it is BOTH (a) at/above a configured **severity floor** AND (b) **deterministically-scored**. *(Hardened correction: determinism-provenance is NOT a stored `state.json` field — `expected_by` lives only in the fixture `EXPECTED_FINDINGS.yaml`. The loop derives provenance from a lint-pinned `slug → deterministic|agent` table seeded from SPEC_1.4.0 §12, so the gate reads a real value, not a phantom column.)* This applies ADR 0004's load-bearing invariant — an agent opinion without deterministic evidence never blocks a human's merge, and by the same principle never auto-files autonomous rework. Agent-scored findings (semantic ADR-contradiction rot, criticality-drift, and the agent-scored TX slugs) stay comment-only and never trigger a drain.
3. **Formatter — spec-gen's findings-register input class.** The spec tier's v1 spec DECLARES `Findings register (a GAPS_SPEC-style audit register)` as a first-class input (spec-gen-tier-v1.md §2 line 34) — *and its non-goals (line 224) explicitly DEFER the wiring as "input class reserved, future scope."* This is a designed-for extension being built now, not the closing of an accidental omission — the honest framing matters. The loop feeds selected findings, in the register shape the tier already understands, into a `/spec --from-findings` session. spec-gen interrogates it like any Draft-Spec input and REFUSES to finalize an incomplete manifest (HC1), so a finding that cannot become a complete, testable manifest degrades to the review path rather than draining blind.
4. **Drainer — autopilot.** *(Hardened correction: the draft claimed "ONE Story PR straight through." Per ADR 0009 and the VERIFIED autopilot lifecycle, Pickup runs GENERATE which opens a **Runbook PR** publishing the predicted file surface, and the drain then files **one-or-more Story PRs** — one per Story, ADR 0007 — not necessarily one.)* A findings-register Spec that reaches `completeness: complete` carries a valid Verification Manifest, so by ADR 0008 `detect_input_mode.sh` returns STRAIGHT_THROUGH. Autopilot NEVER merges (HC4, VERIFIED). A human reviews once, at the end — the zero-trust posture: verify at the source (spec-gen's refusal gate is load-bearing per ADR 0008), not repeatedly downstream.

**The load-bearing guardrail: escalate-instead-of-drain (ADR 0002 verbatim).** A deterministic finding names a real defect, but its *remediation* may be values-laden. "DARK vital on a money path" is a deterministic finding; *what to emit and whether a duplicate is silently deduped or rejected-and-alerted* is risk appetite (ADR 0002's own illustrative cut). The loop classifies not the finding but the **fix** against ADR 0002's trilist, using a deterministic **slug → remediation-class** table (vendored and lint-pinned like the escalation block itself):
- **drain-class** slugs — reversible-at-low-cost AND verifiable downstream by the suite's own gates: `marker, dead-code, commented-code, suppression, memory-rot-dangling-ref, giant-file, test-skip`, vacuous-test, missing-behavior-binding. These route to a drain.
- **escalate-class** slugs — values-laden/irreversible/external-fact: the money/auth/tx slugs verbatim from `audit-state-and-verify.md` + `security/*` + wire-format/public-API/alert-seam. These file a manifest-less GENERATE+pause to spec-gen's S5 escalation — one question at a time, finding as recommendation, dissent attached.

**Hardened honesty note (Defect H).** Because the eligibility gate (step 2) filters agent-scored slugs to INELIGIBLE, several escalate-class TX slugs that are agent-scored in SPEC_1.4.0 §12 (`non-idempotent-handler, unsafe-retry, missing-compensation`) never reach the classifier at all — for those the loop's real behavior is comment-only (ADR 0004-correct), NOT loop-escalation. The escalate-class table stays a superset as a fail-safe, but the loop does NOT claim to autonomously escalate agent-scored money-path fixes; only deterministically-scored escalate-class slugs (`dark-money-movement`, `log-only-refund`-seed) route to the loop's S5.

The table's membership is agent-decidable at the margins but its EXISTENCE and the ADR-0002 rule it encodes are not; the rule is vendored byte-identical (VERIFIED `vendored:escalation-criterion` block in all three tiers) and the loop consults it, never re-derives it.

## Considered Options
- **A fifth `remediation` plugin with its own detectors and marketplace entry** (adversarial position): a cleaner standalone install story. Rejected: it would need its own self-test, vendored manifest schema, vendored escalation block, and marketplace registration — the exact parallel-infrastructure the suite exists to avoid, and it would necessarily re-implement finding selection or severity, creating the C-class contradiction the suite kills. V6 (VERIFIED exactly-four lint) would have to grow. The install-story win is already covered: codebase-health is independently installable and the loop is one of its skills, mirroring how ADR 0003 made the PR Gate a *mode* of the audit.
- **Read `expected_by` from `state.json` for the deterministic gate** (the draft's original mechanism): rejected as non-buildable — VERIFIED that `expected_by` is a fixture-only annotation absent from the runtime findings record. Replaced with the lint-pinned slug-provenance table, which reads a value that actually exists and is drift-checked against SPEC_1.4.0 §12.
- **Drain every above-threshold finding, escalate nothing** (fastest autonomy): rejected — autonomously writes code for silent-dedupe-vs-reject decisions ADR 0002 reserves for a human.
- **Escalate everything, drain nothing** (safest): rejected — re-introduces the rubber-stamp latency ADR 0002 abolished; the low-cost-reversible class genuinely IS agent-decidable and draining it is the loop's entire value.
- **Compute the trigger threshold from agent-scored findings too**: rejected — violates ADR 0004; model-judgment feeding model-judgment with no deterministic evidence and no human between (loop-safety convergence rule, VERIFIED).

## Consequences
- spec-gen SKILL.md's invocation table gains the findings-register row its v1 spec §2 reserved — building the designed-for door (not inventing scope, but also not pretending it was an oversight).
- The severity floor and the escalate-class slug-table membership are per-repo config with shipped defaults (floor = HIGH; the money/auth/wire-format slug set), following ADR 0004's precedent that soak-window and blocking-class list are per-repo config.
- The drain produces a Runbook PR + one-or-more Story PRs (ADR 0007/0009), none merged — the review unit is the Story, as CONTEXT.md defines it, not a single PR.
- The suite closes the ADLC loop end-to-end (findings→spec→drain→PR) without any tier gaining a new opinion; the loop's only authority is *routing*, every routing decision a deterministic slug/severity read or the ADR 0002 rule applied verbatim.
- Promoted to an ADR because it is surprising-without-context (a loop that could be a plugin deliberately is not), hard to reverse (it defines the seam three tiers wire into), and a real trade-off (autonomy vs the escalation guardrail).
