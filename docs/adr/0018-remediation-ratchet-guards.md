# ADR 0018 — The remediation ratchet: three named guards against re-filing, remediating-a-remediation, and infinite regress, all on the existing fingerprint/WONTFIX/tombstone substrate

---
status: agent-decided
date: 2026-07-06
---

A loop that turns findings into rework can eat itself three ways. Each gets a named, deterministic guard reusing the audit tier's EXISTING state substrate (`audit/state.json` per-finding records, the stable `sha1(path:symbol:slug)` fingerprint, the WONTFIX terminal status, and the manifest tombstone `lifecycle: withdrawn`) — no new ledger, no second source of truth (VERIFIED substrate).

**Guard 1 — Never re-file the same finding (idempotency by fingerprint).** The fingerprint IS the loop's idempotency key. Before filing, the loop stamps the finding's `state.json` record with an ADDITIVE `remediation: { status: SPEC_OPEN | PR_OPEN | ESCALATED | WONTFIX, ref, opened_at, remediation_depth }` object (schema stays v2 — additive fields are not a break, VERIFIED precedent). *(Hardened scope: this additive object is the ONE mutation the loop makes to `state.json`; it never touches `status`/`severity`/`verified_by` — those stay `/audit`/`/verify`-owned — so the loop cannot manufacture a false closure.)* A fingerprint already carrying an open record (live spec session, open Story PR, pending S5) is **skipped, loudly** (invariant 6 — into Not-covered/`[skip]`, VERIFIED convention). A human WONTFIX (existing `/verify --wontfix`, VERIFIED) permanently silences it for the loop too. This extends the audit's own convergence guarantee — "a gap found once can never silently recur" — so a *filed* finding is never re-filed.

**Guard 2 — Never remediate a remediation forever (depth ceiling).** The Story PR the loop files is a diff the PR Gate audits like any other. A finding discovered *in* a remediation-authored branch is detectable deterministically: the branch is under the loop's own `remediation/<slug>/*` namespace, using the VERIFIED `claim_overlap.sh --self-namespace <prefix>` → EXCLUDED primitive (claim_overlap.sh:26,107) — *(hardened citation: the draft mis-attributed this to "AV3-09"; the actual primitive is the `--self-namespace` flag)*. Such a finding inherits `remediation_depth = parent+1`, and **`remediation_depth >= 1` is never auto-drained — it escalates**, regardless of slug class. A fix that itself trips the gate is a signal the automated fix is wrong or the finding is deeper than a mechanical remediation reaches — a human's call. This bounds recursion at depth 1 by construction.

**Guard 3 — Never chase its own tail; the ratchet makes inherited debt invisible.** Structural properties, all inherited free:
- The loop **never merges** (autopilot HC4, VERIFIED). It cannot close its own PR and immediately re-audit the merged result within one cadence, so it cannot spin.
- The ADR-0004 ratchet compares NEW debt against the base and makes pre-existing debt structurally invisible (`check_new_debt.sh` added-lines-only). A remediation touching file X cannot resurface X's *inherited* findings as "new," so the loop cannot oscillate on brownfield debt it isn't introducing.
- The loop fires on the shared **CI-cron cadence** and each fire is **stateless and idempotent** — it re-derives eligible findings from `state.json` and open remediation PRs from the host adapter (`pr-list-ready`, ADR 0009) — so a missed or overlapping fire is harmless (the Marshal's property).

**Tombstone interaction — HARDENED (Defect C: authored by the right writer).** When a drain-class remediation removes a symbol (dead-code removal), the `lifecycle: withdrawn` + `withdrawn_reason` tombstone that CH-05's rot facet depends on (VERIFIED `check_memory_rot.sh`) must be written **by spec-gen in the register session's manifest**, NOT by the autopilot Story PR — because spec-gen is the manifest's ONLY writer (VERIFIED HC3 / MS §7) and autopilot writes product code, never the manifest. Concretely: the removal register declares the withdrawal as an acceptance behavior, spec-gen emits the tombstone into the manifest it owns, and the tombstone lands in the Spec/manifest PR. CH-05 then cannot flag the loop's own cleanup as memory-rot. *(This corrects the draft, which incorrectly had the autopilot drain write the tombstone — a wiring bug that would either violate the one-writer contract or leave the removal untombstoned and trip CH-05 on the loop's own cleanup.)* The removed defect's fingerprint moves to WONTFIX-by-removal only with a FIX_LOG entry (VERIFIED `audit-state-and-verify.md` rule; a bare `git log` hit never rounds up).

## Considered Options
- **A new remediation-ledger file tracking filed findings** (adversarial position): explicit, inspectable. Rejected: it duplicates `state.json`'s per-finding identity and would drift — a second source of truth the audit tier's whole design avoids.
- **Have autopilot write the withdrawal tombstone in the Story PR** (the draft's original wiring): rejected — VERIFIED that spec-gen is the manifest's ONLY writer (HC3); autopilot writing a manifest tombstone violates the one-writer contract. The withdrawal is declared in the register spec-gen session instead.
- **No depth ceiling; let the ratchet alone bound recursion**: rejected — the ratchet stops oscillation on *inherited* debt, but a genuinely-new defect introduced BY a remediation is "new debt" the loop would happily re-file; without Guard 2 that is an unbounded fix-a-fix regress.
- **Auto-merge the Story PR to shorten the loop**: rejected outright — violates HC4 and the human-review requirement, and re-opens the tail-chasing failure Guard 3 closes.
- **Fire on every PR-Gate run rather than on cadence**: rejected as default — couples remediation latency to human PR volume and risks a thundering herd of spec sessions; cadence + statelessness is the proven-boring choice. (A repo owner may wire an on-PR trigger; it is idempotent either way.)

## Consequences
- The loop's entire state footprint is a single additive `remediation` object on existing `state.json` records (schema stays v2) plus its own PR namespace on the host. No new persistent store.
- `remediation_depth >= 1 → escalate` is deterministic and self-testable (branch-namespace read via `--self-namespace`), so it lands as a `[det]` assertion, not a `[drain]` residual.
- Corrupt/unreadable `state.json` degrades the loop to **doing nothing** (invariant 4: broken state → less action) — it never files rework it cannot idempotency-check.
- Promoted to an ADR because the three guards are the loop's safety spine (hard to reverse without re-opening a regress), surprising-without-context, and a real trade-off (each guard trades autonomy for a bounded, auditable loop).
