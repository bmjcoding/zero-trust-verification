# ADR 0019 — Org-Wide Memory is a read-only, memory-glob-bounded index/crawler over repo-resident memory, exposed via refuse-by-default MCP — never a second store of truth

---
status: agent-decided
date: 2026-07-06
---

**Governance (Bailey 2026-07-06): the index respects repo visibility.** An agent working in repo A receives memory only from repos the caller identity can already access; private/restricted-repo ADRs, decision-log lines, and manifest excerpts NEVER cross confidentiality boundaries. ACLs are enforced at query time against the caller's identity — the crawler may index broadly, but the MCP query surface refuses to surface a record the caller cannot already read at its source. This is a load-bearing requirement for a regulated org, and it composes with the refuse-by-default MCP posture below.

The suite already produces org memory BY CONSTRUCTION — every tier commits its knowledge into the repo it works on: CONTEXT.md glossaries (canonical terms + rejected `_Avoid_` aliases), `docs/adr/` ADRs (H1 title then post-title YAML frontmatter with `status` ∈ {accepted, agent-decided}; decisions + dissent in Considered Options), the Verification Manifest (journeys with criticality, behaviors, vitals), `DL-###` decision-log lines (PR bodies, trackers, manifest `interrogation.log`), and as-built/journey docs. These formats are already aggregation-ready (stable kebab names, machine-readable frontmatter, a single-JSON-Schema-validated manifest, one-fact-per-file) — and the spec-gen tier already RESERVES `org-memory aggregation (formats already comply)` as future scope. Org-Wide Memory is therefore an **INDEX and CRAWLER OVER these repo-resident files, exposed to agents via a read-only MCP server — never a second store of truth.** Every index record is a derived view carrying a `{repo, commit_sha, path, line}` back-pointer to the authoritative bytes; there is no write-back, no edit surface, no copy of any fact that outlives one crawl. Rebuild-from-empty is always correct. The north star is operational: *if an agent has to be told something twice, we've failed.*

**Why index-not-store.** A second store of truth re-opens the exact failure class the suite exists to prevent: two copies of one fact that drift are a coin-flip at runtime (the C-class contradiction; ADR 0001's vendoring lint, ADR 0003's Memory Rot). A store would need a sync protocol and a staleness policy and would be authoritatively WRONG the moment a repo merged a superseding ADR or bumped `manifest_revision`. By being a derived cache keyed on `commit_sha`, OWM inherits the freshness the repos ALREADY solve instead of re-litigating it. Staleness is disclosed on every answer (`possibly_stale` when the indexed sha lags head, UNKNOWN-safe — absent — when head is unknown; never a false 'fresh'), never hidden.

**Reuse claims, corrected (honesty first — this ADR retracts an r1 overstatement).** OWM sits on proven substrate, but NOT all of it was pre-solved, and the draft that preceded this ADR overstated one dependency. Corrected ground truth:
- `scripts/validate_manifest.py` + the single JSON Schema (ADR 0014) parse manifests, exit-code contract 0/3/4/5 — OWM REUSES that path and honors exit 4/5 as `unparseable`, never forking a parser. (Real reuse.)
- `lint_consistency.sh` gains rule V7 pinning OWM's recognizers to the canonical formats; `suite_self_test.sh` + `SUITE_STRICT` provide the green-run + planted-drift RED-test + skip-honesty discipline; shell + Python-on-uv with bash-3.2/BSD portability is the substrate (ADR 0015). (Real reuse.)
- **`host.sh` does NOT already solve org enumeration.** Verified against `plugins/autopilot/scripts/host.sh` + ADR 0013: host.sh is a 104-line PR/build pass-through (`pr-open … pr-merge build-status pr-list-ready backend`) with NO repo enumeration, NO org listing, NO repo-content read, and it lives inside the autopilot plugin. Org enumeration is therefore a genuinely NEW backend method (`repo-list`), added behind the adapter against the T01-class matrix — described as new work (ADR 0013 pattern: a new capability is a backend method, never a new caller path), never as pre-solved. And because coupling OWM to the autopilot plugin at runtime would destroy independent installability (the ADR 0011/0001 failure class), OWM is CONFIG-FIRST: it crawls an explicit repo-list by default and treats host-enumeration as an OPTIONAL, vendored-under-lint enrichment it degrades without.

**Bounded read surface (feasibility/cost).** OWM reads ONLY a closed allow-list of memory globs (`docs/adr/*.md`, `CONTEXT.md`, the manifest glob, journey/as-built globs, decision-log carriers) — NEVER the code tree, never an arbitrary recursive walk. An unbounded per-repo tree walk across an org is the cost failure this bound exists to prevent (the org-memory analog of never running a bounded probe against the whole suite). A per-repo file-count + byte ceiling yields a loud `memory-surface-oversized` crawl_error rather than a silent hang. OWM has strictly LESS read surface than the audit tier, which reads the whole tree.

**Consumption is refuse-by-default MCP.** Agents are the primary consumer, so the query surface is an MCP server exposing lookup / full-text search / alias-resolution / decision-lookup tools — a thin adapter over a deterministic `query.sh` (the retrieval source of truth), mirroring how `validate_manifest.sh` wraps the `.py`. READ-ONLY tools, no mutation. Two safety invariants: (a) the tools REFUSE-BY-DEFAULT any repo outside the configured allow-list — the enforcement point for a cross-repo disclosure policy that is a human call (escalated); (b) SELF-EXCLUSION — OWM never indexes its own emitted artifacts, so no citation loop where it re-ingests and re-serves its own derived view can form. The CLI answers even when the MCP server is absent (degrade-gracefully). There is NO remediation loop, NO re-plan, NO write path — so infinite-regress / self-remediation is structurally absent, not merely guarded.

## Considered Options

- **A synced org-memory store (the rejected default)** — a service/DB repos push memory into. Rejected: re-introduces the sync problem + a staleness policy the repos already solve, and becomes authoritatively wrong on every superseding ADR / manifest bump — the C-class contradiction the vendoring lint and Memory Rot machinery exist to kill. An index over source files has no such class: worst case it is a rebuild behind.
- **Claiming host.sh already solves enumeration (the r1 position)** — rejected as false. host.sh is PR/build-scoped and autopilot-resident; asserting reuse there would have shipped a design that either doesn't run (no repo-list exists) or silently couples OWM to the autopilot plugin (breaks ADR 0011 independent installability). The honest design makes enumeration new, optional, config-first, and vendored-under-lint.
- **An unbounded crawler that greps every file per repo** — rejected on cost: across an org this is the perf/cost blow-up the bounded memory-glob surface (OWM-03a) prevents. The read surface is a closed allow-list with a per-repo ceiling.
- **A vector/embedding store as the v1 primitive** — deferred, not dismissed. FTS5 over single-file SQLite is the v1 baseline; a semantic layer sits BEHIND the query surface (never a new caller path) with a stated revisit trigger (FTS recall becomes a documented adoption blocker) — mirroring ADR 0015's Rust deferral. Unproven cost vs unproven need.
- **Bolt the crawler onto an existing plugin (audit or autopilot)** — rejected: forces OWM's enumeration + MCP surface onto installers who want only that plugin, collapses the per-plugin permission surface, and re-opens the release-integrity failure class ADR 0001/0011 closed. A fifth independently installable, repo-READ-only plugin keeps the permission surface honest (OWM read-only; audit warn-only; autopilot and Marshal hold the write surface).
- **An MCP surface that serves any indexed repo regardless of scope** — rejected: cross-repo disclosure (can an agent in repo A see repo B's private ADRs and decision log?) is a data-governance call, not agent-decidable. Refuse-by-default is the safe posture until a human sets the policy (escalated).
- **A semantically-aware freshness heuristic (TTLs)** — rejected: any independent invalidation heuristic is a second thing to drift. The commit-sha cache-key is the entire staleness story; freshness beyond 'indexed sha vs head' is disclosed as UNKNOWN, never guessed.

## Consequences

- OWM is the FIFTH plugin. The root marketplace `description` and `lint_consistency.sh` V6 (whose python3 structural check HARDCODES the expected four-plugin set and refuses a rogue fifth) move from four to five: the `expected` dict grows, name↔source pairing is verified, and V6's swap/rogue red-tests extend so a rogue SIXTH now trips RED. `suite_self_test.sh` gains an `org-memory` component and the V7 planted-drift red-test.
- `host.sh` gains a NEW `repo-list` backend method implemented by both backends against the T01-class mock matrix (ADR 0013) — landing either as an autopilot-tier item or vendored into OWM under the ADR 0001 byte-identity lint; OWM takes NO runtime dependency on the autopilot plugin.
- OWM depends on the manifest JSON Schema + the `## Behavior coverage` format staying single-sourced; V7 makes a format change that would silently break OWM's parse a caught cross-plugin change, not a runtime surprise.
- The index is a disposable single-file SQLite cache: rebuild-from-empty is the recovery path; a schema_version bump forces a rebuild (no migrations framework).
- The MCP surface is refuse-by-default on out-of-scope repos and self-excludes OWM's own outputs; there is no write path, no remediation loop, no mutation testing — nothing that could form an autonomous regress.
- Genuinely human/infra decisions do NOT block the build: WHERE the index runs and on what cadence; WHICH identity enumerates the org and its READ scope; and WHETHER private/restricted-visibility repo memory may be surfaced cross-repo — are escalated (human_questions). The design is formats-ready and config-first so OWM-01..08 land and self-test hermetically against fixtures regardless of the hosting/auth/disclosure answers.
